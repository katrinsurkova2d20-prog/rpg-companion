# TODOFIRST — Живой архитектурный ориентир проекта

Этот файл — **первое место**, куда смотрим перед любой задачей.
Цель: развивать проект поэтапно, не останавливая фичи, но при этом постоянно улучшая архитектуру.

---

## 1) Основной принцип

**Разделяем UI, правила, данные и состояние.**

- Экран/модалка = отображение + пользовательские события.
- Бизнес-логика = отдельные функции/модули (без UI).
- Данные контента = JSON-каталоги.
- Состояние и сохранение = отдельный слой.

> В любой новой задаче сначала проверяем, не смешивается ли UI с игровой логикой.

---

## 2) Целевая структура (ориентир)

```txt
components/
  screens/                 # Только композиция UI экранов
    <ScreenName>/
      <ScreenName>.js
      modals/              # UI модалок только этого экрана
      parts/               # Локальные подкомпоненты UI (если нужно)
  common/                  # Переиспользуемые UI-компоненты

styles/                    # Стили по фичам/экранам
  character/
  inventory/
  weapons/

assets/                    # Картинки, иконки, медиа

data/                      # JSON-данные игры
  traits/
  origins/
  perks/
  weapons/
  armor/
  items/
  mods/
  equipmentKits/

i18n/                      # Переводы
  <locale>/
    screens/
    data/                  # Переводы для сущностей из data/

domain/                    # Чистая игровая логика (без UI и без БД)
  character/
  traits/
  perks/
  equipment/
  effects/

application/               # Сценарии/юзкейсы (оркестрация)
  character/
  inventory/
  equipment/

state/                     # Context/store/selectors/actions

infrastructure/            # БД, адаптеры, sync, внешние сервисы
```

**Важно:** это целевая схема. Переходим к ней постепенно, по мере новых задач.

### Именование файлов данных (человеко-ориентированно)

Чтобы было удобно работать мастеру игры, имена файлов должны сохранять контекст, а не быть абстрактными:

- Для черт: `<originName-traitName-id>.json`
- Для комплектов: `<originName-kitName-id>.json`

Примеры:

- `vaultdweller-goodsoul-01.json`
- `brotherhood-combatstarter-02.json`

Правило:

- `originName` и `traitName/kitName` — понятные и стабильные сегменты (предпочтительно латиница/транслит в `kebab-case` или `snake_case`).
- `id` — короткий суффикс уникальности (`01`, `02`, `a1`), если есть похожие варианты.
- Не используем абстрактные имена вида `trait-2.json`, `kit-7.json`.

Отдельно фиксируем соглашение:

- **Ключи и переменные в коде**: `camelCase` (как базовый стиль проекта).
- **Названия файлов data-сущностей**: через разделители `-` или `_`, чтобы визуально проще читать составные имена.

Это сделано специально, чтобы по имени файла сразу видеть происхождение и сущность, даже без открытия JSON.

## 3) Непрерывный рефакторинг по фичам

Работаем по правилу: **каждая новая фича должна слегка улучшать структуру.**

### На любую задачу (черта, перк, оружие, броня, мод, предмет):

1. Добавить/нормализовать данные в `data/`.
2. Добавить/обновить правила в `domain/`.
3. Добавить/обновить use-case в `application/`.
4. Подключить в `state/` (если нужно).
5. В `components/screens/*` оставить только UI-часть.
6. Добавить i18n-ключи для UI и data.
7. Проверить обратную совместимость (минимум ручной smoke-check).

---

## 4) Шаблоны задач

### 4.1 Добавить новую черту

- `data/traits/<originName-traitName-id>.json`: id, условия, эффекты, совместимости/исключения.
- `domain/traits/`: функция применения/снятия эффекта черты.
- `application/character/`: сценарий выбора/смены черты.
- UI: модалка выбора и отображение результата (без расчетов в JSX).
- i18n: название/описание/тексты ошибок и ограничений.

### 4.2 Добавить комплект снаряжения для черты/происхождения

- `data/equipmentKits/<originName-kitName-id>.json`: список предметов, количества, условия.
- `domain/equipment/`: резолвер комплекта и проверка ограничений.
- `application/equipment/`: сценарий применения комплекта к персонажу.
- UI: только выбор и подтверждение.

### 4.3 Добавить оружие/броню/моды

- Данные в `data/weapons`, `data/armor`, `data/mods`.
- Правила влияния в `domain/equipment`.
- Логику применения и пересчёта — в `application/equipment`.
- Экран инвентаря/экипировки только вызывает use-case и рендерит результат.

### 4.4 Добавить предметы/расходники/эффекты

- `data/items` (+ тип эффекта/длительность/условия).
- `domain/effects` для применения и истечения эффектов.
- `application/character` для обновления состояния персонажа.

---

## 5) Правила влияний (“всё влияет на всё”)

Чтобы не утонуть в хаосе, применяем единый формат модификаторов.

Каждый эффект должен иметь:

- `source` — откуда эффект (trait/perk/item/mod/condition).
- `target` — что изменяем (атрибут, навык, урон, защита и т.д.).
- `operation` — тип операции (`add`, `mul`, `set`, `min`, `max`).
- `value` — значение операции.
- `conditions` — когда активен эффект.
- `priority` — приоритет применения.

### Порядок применения эффектов (фиксированный)

1. Базовые значения персонажа.
2. Происхождение/черты.
3. Экипировка и моды.
4. Перки.
5. Временные эффекты/состояния.
6. Ограничения и кэппинг.

Если новая логика ломает этот порядок — сначала фиксируем порядок, потом добавляем фичу.

---

## 6) Definition of Done для любой фичи

Фича считается завершённой только если:

- [ ] Данные лежат в правильном разделе `data/`.
- [ ] Правила лежат в `domain/`.
- [ ] Сценарий применения лежит в `application/`.
- [ ] UI не содержит бизнес-вычислений, кроме простого форматирования.
- [ ] Добавлены i18n-ключи.
- [ ] Проверены базовые сценарии (ручные или тесты).
- [ ] Нет дублирующей старой логики без TODO/плана удаления.

---

## 7) Антипаттерны (так не делаем)

- Не добавляем длинные расчёты в `Screen`/`Modal` компоненты.
- Не храним “магические” правила прямо в JSX.
- Не копируем один и тот же алгоритм в несколько экранов.
- Не добавляем новую сущность только в UI без data/domain слоя.
- Не смешиваем перевод UI и перевод data в случайных местах.
- Не вводим непрозрачные названия файлов/констант без контекста происхождения/сущности.

---

## 8) Язык и человеко-понятные алиасы

Так как разработка и геймдизайн ведутся в основном на русском, фиксируем правило двойного представления:

- В коде — стабильные технические ключи в `camelCase`, константы в `UPPER_SNAKE_CASE`.
- В именах файлов `data/*` — составные части через `-` или `_` (например `origin-trait-id.json`), чтобы не терять читаемость.
- В `i18n/ru-RU/*` — человеко-понятные русские названия и описания.
- Между ними должна быть явная связь через ключи (`id`, `slug`, `code`), чтобы можно было легко искать “русский термин ↔ технический ключ”.

Рекомендуется добавлять служебные поля:

- `displayNameKey` — ключ перевода названия.
- `descriptionKey` — ключ перевода описания.
- `aliases` (опционально) — список привычных имен/синонимов для поиска в инструментах и отладке.

## 9) Как использовать этот файл в реальной работе

Перед стартом задачи:

1. Открыть `TODOFIRST.md`.
2. Определить тип задачи: trait/perk/equipment/item/weapon/armor/mod.
3. Пройти соответствующий шаблон из раздела 4.
4. Проверить DoD из раздела 6.
5. В PR/коммите указать, какой маленький шаг к целевой структуре сделан.

---

## 10) Принцип поэтапной миграции

Мы **не делаем большой “переписон” сразу**.

Мы делаем:

- маленькие безопасные шаги,
- только в рамках текущей фичи,
- с сохранением работоспособности,
- с постепенным переносом логики из экранов в `domain/application`.

Это основной рабочий режим проекта до полной стабилизации архитектуры.
